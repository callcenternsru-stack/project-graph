1. Введение

Проект представляет собой комплексную экосистему для управления колл-центром. Он включает два основных модуля:

  • Модуль учёта рабочего времени – синхронизация через Netlify Blobs, функции getData/saveData.
  • Модуль рекрутинга – полностью переведён на серверную архитектуру: данные хранятся в Netlify Blobs, доступ осуществляется через 38 Netlify Functions. localStorage не используется.

2. Архитектура системы

Система построена на статических HTML-страницах, JavaScript (клиентская логика) и Netlify Functions для работы с хранилищами. Все данные (кроме временных сессий) хранятся в Netlify Blobs в нескольких отдельных хранилищах. Аутентификация рекрутеров происходит через функцию loginRecruiter с проверкой пароля, сессия сохраняется в sessionStorage. Партнёры (модуль учёта времени) также используют sessionStorage.

3. Настройка окружения

Для разработки необходим любой современный браузер. Для работы с Netlify Blobs требуется аккаунт Netlify и настроенные переменные окружения NETLIFY_SITE_ID и NETLIFY_ACCESS_TOKEN. Подробнее – в разделе 9.

5. Формат данных в Blobs

Используются следующие хранилища Netlify Blobs:

  • app-data – для модуля учёта времени (проекты, месяцы, сотрудники, записи, партнёры, комментарии).
  • candidates-data – для автоматических анкет (ключ – уникальный код кандидата, значение – объект с formData, статусом, ссылками на файлы).
  • candidates – для кандидатов, добавленных рекрутерами.
  • recruiters – учётные записи рекрутеров (ключ – имя рекрутера).
  • projects, callResults, contactTypes – справочники.
  • scripts – скрипты для рекрутеров.
  • projectMeetings, projectTrainings, projectMeetingComments, projectTrainingComments, scheduleMonths – данные для графиков.
  • manualForms – ручные анкеты.

6. Серверные функции (Netlify Functions)

Все функции написаны на JavaScript (Node.js) и используют библиотеку @netlify/blobs. Перечислим основные группы.

6.1 Учёт времени

  • getData – загружает состояние из хранилища app-data.
  • saveData – сохраняет полное состояние учёта времени.

6.2 Рекрутеры и аутентификация

  • getRecruiters – список всех рекрутеров.
  • saveRecruiter – создание/редактирование рекрутера.
  • deleteRecruiter – удаление рекрутера.
  • loginRecruiter – проверка логина и пароля, возврат данных рекрутера (без пароля).

6.3 Кандидаты от рекрутеров

  • getCandidates – все кандидаты (для HR).
  • getMyCandidates – кандидаты конкретного рекрутера.
  • submitCandidate – сохранение/обновление кандидата.

6.4 Справочники

  • getProjects, saveProject, deleteProject – проекты.
  • getCallResults, saveCallResult, deleteCallResult – результаты звонков.
  • getContactTypes, saveContactType, deleteContactType – типы обращений.

6.5 Скрипты

  • getScripts, saveScript, deleteScript – скрипты для рекрутеров.

6.6 Данные графиков

  • getProjectMeetings, saveProjectMeetings – организационные встречи.
  • getProjectTrainings, saveProjectTrainings – обучения.
  • getProjectMeetingComments, saveProjectMeetingComments – комментарии к встречам.
  • getProjectTrainingComments, saveProjectTrainingComments – комментарии к обучениям.
  • getScheduleMonths, saveScheduleMonths – список месяцев графика.

6.7 Автоматические анкеты и результаты

  • submitForm – сохраняет анкету и генерирует запись со статусом pending.
  • checkCode – проверяет, существует ли код и имеет ли он статус pending (используется программой).
  • submitResults – принимает multipart/form-data с файлами (report.txt, results.json, voice_recording.wav) и кодом, сохраняет файлы и обновляет статус на completed.
  • getResults – возвращает список завершённых анкет со ссылками на файлы.
  • getFile – отдаёт файл по параметрам code и file (например, ?code=...&file=report.txt).

6.8 Ручные анкеты

  • submitManualForm – сохраняет ручную анкету.
  • getManualForms – возвращает все ручные анкеты.

7. Клиентские страницы

7.1 index.html – админ-панель учёта времени

Полнофункциональная панель для администратора: управление проектами, сотрудниками, партнёрами, просмотр графиков и статистики. Синхронизация с Netlify Blobs через getData и saveData.

7.2 partner.html – панель партнёра

Аналог админ-панели, но с ограниченным доступом – партнёр видит только свои проекты и своих сотрудников. Вход по логину/паролю из массива partners, хранящегося в Blobs. Данные синхронизируются с сервером.

7.3 view.html – публичный просмотр

Страница только для чтения, позволяет любому пользователю выбрать проект и месяц и увидеть график работы. Данные загружаются через getData.

7.4 recruiter.html – панель рекрутера (с входом)

Содержит форму входа (логин/пароль). После успешной аутентификации через loginRecruiter загружает:

  • проекты, результаты звонков, типы обращений;
  • скрипты;
  • список кандидатов текущего рекрутера.

Позволяет просматривать скрипты, редактировать и добавлять кандидатов (сохраняются через submitCandidate). Данные хранятся только на сервере, localStorage не используется.

7.5 hr control panel.html – админ-панель рекрутинга

Основная панель для управления рекрутингом. Загружает все данные через соответствующие get* функции. Вкладки:

  • Рекрутеры – CRUD для учётных записей рекрутеров.
  • База контактов – список кандидатов, добавленных рекрутерами.
  • Скрипты – редактирование скриптов.
  • График – планирование встреч и обучений (добавление месяцев, событий, комментариев).
  • Все анкеты – заглушка.
  • Анкеты авто – отображает завершённые автоматические анкеты из getResults со ссылками на отчёты (через getFile).
  • Анкеты ручные – отображает анкеты из getManualForms.

Все изменения сохраняются через соответствующие save* и delete* функции.

7.6 candidate.html – анкета кандидата (проект РОССЕТТИ)

Страница для сбора первичных данных кандидата. Содержит информацию о проекте, FAQ, форму с обязательными полями. Две кнопки:

  • Пройти проверку автоматически – генерирует уникальный код, отправляет данные формы и код через submitForm на сервер, затем показывает блок с кодом и ссылкой на скачивание SystemCheckPRO_Setup.exe (установщик программы).
  • Пройти проверку вручную – отправляет данные формы через submitManualForm и показывает инструкцию по отправке скриншотов.

8. Аутентификация и сессии

Партнёры (учёт времени): проверка логина/пароля из массива partners, хранящегося в Blobs. Сессия сохраняется в sessionStorage.

Рекрутеры: вход по логину/паролю через функцию loginRecruiter. При успехе данные рекрутера сохраняются в sessionStorage. Выход очищает сессию.

9. Инструкция по развёртыванию на Netlify

  1. Собрать все файлы в структуру, описанную в разделе 4.
  2. Убедиться, что в package.json есть зависимости @netlify/blobs и busboy.
  3. Скомпилировать SystemCheckPRO.ps1 в EXE и поместить в папку public вместе с установщиком.
  4. Загрузить репозиторий на GitHub и подключить к Netlify.
  5. В настройках Netlify указать:
    • папку публикации public;
    • папку функций netlify/functions.
  6. Добавить переменные окружения NETLIFY_SITE_ID и NETLIFY_ACCESS_TOKEN (токен из UI Netlify).
  7. Выполнить деплой.

После деплоя все функции будут доступны, страницы рекрутинга работают с сервером.

10. Локальное тестирование (без сервера)

Для локального тестирования можно открыть HTML-файлы в браузере, но для работы с функциями потребуется эмуляция (Netlify CLI) или реальный деплой. Модуль учёта времени при локальном открытии будет сохранять данные в localStorage (без синхронизации).

11. Возможные проблемы

  • Ошибки 404 при вызове функций – проверьте путь в netlify.toml (functions = "netlify/functions").
  • Ошибки 500 в функциях – смотрите логи функций в панели Netlify; убедитесь, что переменные окружения заданы.
  • Файлы не открываются – проверьте, что все HTML лежат в папке public и publish = "public".
  • Программа SystemCheckPRO не отправляет результаты – проверьте лог debug_submit.log в папке установки; убедитесь, что код действителен и не использован ранее.
  • При скачивании из HR-панели ошибка File not found – исправлено в версии 6.1 (см. раздел 13.6).
  • Speedtest завершается с ошибкой – проверьте, что используется актуальная версия speedtest.exe и включены флаги --accept-license --accept-gdpr (исправлено в версии 6.2).

13. Программа автоматической проверки кандидата (SystemCheckPRO.exe)

13.1. Назначение и возможности

Автономная программа для Windows, которая собирает информацию о компьютере, проводит тест скорости печати, записывает голос и измеряет скорость интернета.

13.2. Собираемые данные

  • Характеристики ПК (процессор, ОЗУ, диск, ОС).
  • Тип подключения к интернету (LAN/Wi-Fi).
  • Анализ аудиоустройств (тип подключения, статус, назначение).
  • Тип устройства (ноутбук/ПК).
  • Тест печати: скорость, ошибки, точность.
  • Запись голоса через SoX.
  • Скорость интернета (пинг, загрузка, отдача) до фиксированных серверов (Ижевск, Видное, Кемерово, Москва).

13.3. Использование и распространение

Программа распространяется в виде установщика SystemCheckPRO_Setup.exe, созданного с помощью Inno Setup. Установщик включает все необходимые файлы (sox.exe, speedtest.exe, библиотеки) и создаёт ярлыки в меню «Пуск» и на рабочем столе (по желанию пользователя). Кандидат запускает программу, вводит уникальный код с сайта, проходит тесты. Результаты сохраняются локально и отправляются на сервер.

13.4. Форматы отчётов

  • results.json – детальный JSON-отчёт.
  • report.txt – текстовый отчёт для пользователя и HR.

13.5. Интеграция с веб-частью

Программа взаимодействует с сервером следующим образом:

  • Перед началом тестирования отправляет GET-запрос к /.netlify/functions/checkCode?code=.... Если код не найден или уже использован, программа завершается с сообщением.
  • После завершения всех тестов формирует multipart/form-data POST-запрос к /.netlify/functions/submitResults с полем code и тремя файлами: report.txt, results.json, voice_recording.wav.
  • При успешной отправке показывает финальное окно с просьбой не закрывать программу в течение 10 минут (время проверки) и затем автоматически закрывается.
  • При ошибке отправки данные сохраняются локально, а пользователю предлагается обратиться к администратору.

13.6. Доработки программы (версия 6.1 и 6.2)

В ходе финальной доработки программы были выполнены следующие улучшения:

  • Исправление отправки результатов: из-за отсутствия параметра -Form в старых версиях PowerShell был добавлен C#-класс MultipartUploader, который использует HttpClient и гарантирует работу на всех поддерживаемых Windows-системах.
  • Добавление мастер-ключа: при вводе кода ADMIN123 программа пропускает проверку на сервере и не отправляет результаты, что удобно для тестирования.
  • Логирование: все попытки отправки записываются в файл debug_submit.log, что позволяет диагностировать проблемы.
  • Кнопка пропуска теста печати: временно добавлена для ускорения отладки; при нажатии тест пропускается с нулевыми результатами.
  • Изменение финального окна: при успешной отправке отображается зелёное сообщение: «Все данные успешно отправлены на сервер. Ваши ответы проверяются. Проверка займет до 10 минут, пожалуйста не закрывайте окно.» После завершения всех фоновых процессов программа закрывается автоматически.
  • Промежуточное окно благодарности: после записи голоса появляется окно с текстом «Спасибо за ваши ответы, ожидайте, идет проверка...», которое висит во время выполнения тестов и закрывается перед завершением программы.
