2. Структура и семантика

Файл организован как клиентская библиотека для работы с Netlify Blobs API:

  • Lambda-совместимость: функция connectLambda извлекает параметры из события Lambda (заголовки x-nf-deploy-id, x-nf-site-id и данные из event.blobs) и устанавливает контекст окружения.
  • Класс Store — центральный класс для всех операций с хранилищем:
    • Конструктор принимает client и либо deployID, либо name.
    • Автоматически формирует внутреннее имя хранилища с префиксами:
      • deploy:<deployID>[:<name>] — для деплой-специфичных хранилищ
      • netlify-internal/legacy-namespace/ — для обратной совместимости
      • site:<name> — для сайт-специфичных хранилищ
    • Методы: delete(), get(), getMetadata(), getWithMetadata(), list(), set(), setJSON().
    • Валидация ключей и имён хранилищ (длина, запрещённые символы).
    • Поддержка пагинации через итераторы (getListIterator).
  • Фабрики Store:
    • getStore(input) — универсальная фабрика, принимает строку (имя хранилища) или объект с параметрами (name, deployID, siteID, token, apiURL и др.).
    • getDeployStore(input) — специализированная фабрика для деплой-хранилищ, с поддержкой экспериментальных регионов (context, auto).
  • Листинг хранилищ: listStores(options) возвращает список всех доступных хранилищ сайта (с фильтрацией деплой-хранилищ).

3. Стилизация (CSS) — не применимо

Данный файл является серверным/клиентским JavaScript-модулем и не содержит элементов пользовательского интерфейса.

4. JavaScript (логика и интерактивность)

  • Асинхронность: все операции с хранилищем возвращают Promise и используют async/await.
  • Итераторы: метод list() может возвращать асинхронный итератор (при paginate: true) для постраничной загрузки.
  • Типы данных: поддержка различных форматов данных:
    • text — строка UTF-8
    • arrayBuffer — бинарные данные
    • blob — объект Blob
    • json — автоматический парсинг JSON
    • stream — поток ReadableStream
  • Метаданные: поддержка пользовательских метаданных при установке и получении блобов.
  • ETag и условные запросы: метод getWithMetadata поддерживает if-none-match для кэширования.
  • Валидация:
    • Ключи не могут быть пустыми, не могут начинаться с / или %2F, максимальная длина UTF-8 — 600 байт.
    • Имена хранилищ не могут содержать /, максимальная длина — 64 байта.
    • Deploy ID должен соответствовать формату \w{1,24}.
  • Контекст окружения: глобальный контекст (siteID, deployID, token, edgeURL) устанавливается через setEnvironmentContext или автоматически из переменных окружения Netlify.
  • Обработка ошибок: специфические ошибки BlobsInternalError, MissingBlobsEnvironmentError.
  • Экспериментальные регионы: поддержка выбора региона через experimentalRegion со значениями context (из окружения) или auto.

5. Форма и отправка данных

Библиотека не имеет пользовательских форм. Данные передаются через HTTP-запросы к API Netlify Blobs:

  • PUT — загрузка блоба (тело запроса содержит данные, метаданные в заголовке X-Netlify-Blobs-Metadata)
  • GET — получение блоба (поддержка условных запросов через If-None-Match)
  • HEAD — получение метаданных
  • DELETE — удаление блоба
  • GET с параметрами — листинг (параметры prefix, cursor, directories)

6. Особенности реализации

  • Префиксы хранилищ: библиотека автоматически добавляет префиксы к именам хранилищ для изоляции данных между сайтами и деплоями.
  • Обратная совместимость: поддержка legacy-формата netlify-internal/legacy-namespace/ для миграции.
  • Пагинация: реализована через асинхронные итераторы, что позволяет эффективно обрабатывать большие списки.
  • Кэширование: через ETag и условные запросы.
  • Lambda-интеграция: функция connectLambda позволяет легко интегрировать библиотеку в Netlify Functions.
  • Гибкость конфигурации: поддержка различных способов передачи параметров (переменные окружения, явные параметры, контекст).
  • Кроссплатформенность: работает в Node.js, браузерах и Edge-окружениях благодаря использованию fetch и ReadableStream.
  • Типизация: хотя файл на JavaScript, он используется в связке с .d.ts файлами для TypeScript-поддержки.

7. Потенциальные точки расширения

  • Сжатие данных: автоматическое сжатие/распаковка больших блобов.
  • Шифрование: клиентское шифрование данных перед отправкой.
  • Кэширование на клиенте: реализация локального кэша для уменьшения количества запросов.
  • Batch-операции: массовое удаление или копирование блобов.
  • Поддержка TTL: автоматическое удаление устаревших блобов.
  • WebSocket уведомления: оповещения об изменениях в хранилище.
  • Инструменты миграции: копирование между хранилищами или окружениями.

8. Особенности для тестирования

  • Настройка тестового окружения: необходимо установить переменные окружения: 
    process.env.NETLIFY_BLOBS_CONTEXT = JSON.stringify({
  deployID: 'test-deploy-123',
  siteID: 'test-site-456',
  token: 'test-token',
  edgeURL: 'http://localhost:3000'
});
  • Создание Store: 
    import { getStore } from '@netlify/blobs/dist/main.js';
const store = getStore('my-store');
  • Базовые операции: 
    await store.set('key', 'Hello World');
const data = await store.get('key'); // 'Hello World'
await store.delete('key');
  • Тестирование с локальным сервером: можно запустить BlobsServer из server.js и указать edgeURL на него.
  • Проверка валидации: попытаться создать ключ с недопустимыми символами — должна выбрасываться ошибка.
  • Тестирование пагинации: создать много блобов и проверить, что итератор возвращает все страницы.
  • Тестирование метаданных: 
    await store.set('key', 'data', { metadata: { author: 'John' } });
const { metadata } = await store.getWithMetadata('key');
  • Тестирование условных запросов: 
    const { etag } = await store.getWithMetadata('key');
const result = await store.getWithMetadata('key', { etag }); // status 304
  • Lambda-совместимость: вызвать connectLambda с корректным событием и проверить, что контекст установлен.
