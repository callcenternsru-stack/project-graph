2. Структура и семантика

deno.lock имеет корневой объект со следующими основными полями:

  • version – версия формата lock-файла (в примере "5").
  • remote – объект, содержащий информацию об удалённых модулях. Ключи – URL-адреса модулей, значения – хеши (целостности) этих модулей.
  • workspace – информация о рабочем пространстве. В данном случае содержит packageJson с массивом dependencies, где указаны зависимости из npm, используемые Deno.

В разделе remote представлены модули, загруженные с edge.netlify.com:

{
  "https://edge.netlify.com/": "fd941d61d88673d5f28aab283fb86fcc50f08a3bc80ee5470498fcfa88c65cfb",
  "https://edge.netlify.com/bootstrap/config.ts": "6a2ce0e544e15e8f8883a5c18da5948e37fd0f2619f68cb31f3af53c51817025",
  ...
}

Здесь каждый ключ – это URL модуля, а значение – хеш SHA-256 этого модуля. Это гарантирует, что при повторной загрузке будет использован тот же самый код.

Раздел workspace.packageJson.dependencies указывает, что проект использует npm-пакет @netlify/blobs версии 7 (через npm:@netlify/blobs@7). Это означает, что в edge-функциях можно импортировать этот пакет.

6. Особенности реализации

  • Не редактировать вручную: любые изменения в lock-файле могут привести к несоответствию версий и ошибкам выполнения. Все изменения должны происходить через обновление кода edge-функций и последующий запуск Deno.
  • Хранение в репозитории: файл deno.lock должен быть добавлен в систему контроля версий (Git), чтобы обеспечить воспроизводимость сборок.
  • Связь с edge-функциями Netlify: если в проекте используются edge-функции (файлы .ts в папке netlify/edge-functions), Deno автоматически создаёт и обновляет этот lock-файл.
  • Использование npm-пакетов: через запись npm:@netlify/blobs@7 Deno позволяет использовать npm-пакеты в edge-функциях. Это удобно, если нужно использовать те же библиотеки, что и в Node.js функциях.
  • Версионирование: символ @7 означает, что используется мажорная версия 7 пакета (любая подходящая минорная версия). Точная версия будет зафиксирована при первом запуске.
  • Отличие от package-lock.json: в отличие от npm, Deno работает с URL-импортами, поэтому lock-файл хранит хеши удалённых модулей, а не версии из реестра npm.

7. Потенциальные точки расширения

  • Добавление новых edge-функций: при создании новых edge-функций, использующих дополнительные удалённые модули, deno.lock автоматически обновится.
  • Обновление зависимостей: если потребуется обновить версию @netlify/blobs до 8, нужно изменить код функций и затем запустить их – lock-файл обновится.
  • Использование других npm-пакетов: можно добавить другие npm-пакеты в edge-функции, используя синтаксис npm:package@version. После первого запуска они появятся в deno.lock.
  • Настройка кэширования: можно настроить поведение Deno в отношении lock-файла через переменные окружения или флаги.
  • Интеграция с CI/CD: в CI-средах рекомендуется использовать deno cache --lock=deno.lock для предварительной загрузки зависимостей.

8. Особенности для тестирования

  • Проверка целостности: выполните команду deno cache --lock=deno.lock --lock-write в папке с edge-функциями. Это обновит lock-файл (если нужно) и покажет ошибки, если есть несоответствия.
  • Отсутствие ошибок: убедитесь, что edge-функции, использующие @netlify/blobs, работают корректно (например, локально через Netlify CLI).
  • Проверка версий: можно вручную проверить, что хеши в remote соответствуют ожидаемым (но это редко требуется).
  • Тестирование в изоляции: удалите файл deno.lock и запустите edge-функцию через Netlify CLI. Файл должен создаться заново с теми же хешами (если версии не изменились).
  • Конфликты версий: если после обновления edge-функций возникают ошибки, попробуйте удалить lock-файл и дать Deno создать его заново.
  • Локальное тестирование с Netlify CLI: запустите netlify dev – CLI автоматически обработает edge-функции и использует deno.lock для кэширования.
