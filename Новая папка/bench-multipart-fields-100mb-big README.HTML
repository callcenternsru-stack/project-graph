2. Структура и семантика

Скрипт предназначен для бенчмаркинга и сравнения трёх популярных библиотек для разбора multipart/form-data при работе с файлами. Он генерирует синтетический запрос, состоящий из нескольких файлов с большим объёмом данных (10, 10, 10, 20, 50 МБ), и измеряет время, затраченное каждой библиотекой на полную обработку. Это позволяет оценить производительность при работе с крупными файловыми загрузками.

  • Генерация данных: функция createMultipartBuffers создаёт буферы, соответствующие multipart-сообщению с файлами заданных размеров. Каждый файл имеет имя randomN.bin и заполняется повторяющимися символами '0'.
  • Счётчик событий: объект calls подсчитывает количество вызовов различных этапов парсинга (начало части, заголовок, данные, конец). Это позволяет убедиться, что все библиотеки корректно обработали все файлы.
  • Выбор библиотеки: через аргумент командной строки передаётся имя модуля (busboy, formidable или multiparty). В зависимости от этого выполняется соответствующий блок кода.
  • Измерение времени: перед началом обработки вызывается console.time(moduleName), по окончании (событие close или end) — console.timeEnd(moduleName).

4. JavaScript (логика и интерактивность)

  • Синхронная генерация данных: все буферы создаются заранее.
  • Асинхронная обработка: парсеры работают в событийно-ориентированном стиле, данные передаются через вызовы write() или push().
  • Подсчёт событий:
    • Для busboy используется событие file, внутри которого подписываемся на события потока файла: data (инкрементирует partData) и end (инкрементирует partEnd). partBegin инкрементируется при старте файла.
    • Для formidable используется событие data, где имя события (partBegin, partData, partEnd и т.д.) инкрементирует соответствующий счётчик. Это возможно, потому что formidable генерирует события для каждого шага парсинга.
    • Для multiparty применяется «перехват» (hijack) внутренних методов onParseHeaderField, onParseHeaderValue, onParsePartBegin, onParsePartData, onParsePartEnd для увеличения счётчиков.
  • Обработка больших файлов: размер файлов до 50 МБ, библиотеки должны работать с лимитами, установленными в Infinity (или аналогичными).

5. Форма и отправка данных

Скрипт не отправляет данные по сети; он симулирует получение multipart-запроса с файлами, передавая буферы напрямую парсеру. Это позволяет измерить чистое время парсинга без влияния сетевой задержки и дискового ввода-вывода.

6. Особенности реализации

  • Гранулярный подсчёт событий: для каждого парсера используются разные методы подсчёта, чтобы получить сопоставимые данные о количестве обработанных частей и данных.
  • Бесконечные лимиты: для busboy и multiparty устанавливаются fieldSizeLimit: Infinity и maxFilesSize: Infinity, чтобы избежать обрезания больших файлов.
  • Хитрый перехват в multiparty: поскольку multiparty не предоставляет прямых событий для каждого шага парсинга, скрипт временно заменяет внутренние методы, чтобы вести подсчёт.
  • Использование Readable stream для multiparty: создаётся фиктивный HTTP-запрос (new Readable), в который последовательно передаются буферы.
  • Обработка файлов в busboy: в отличие от полей (field), для файлов busboy предоставляет поток (stream), через который поступают данные. Скрипт подписывается на события этого потока для подсчёта.
  • Отсутствие внешних зависимостей: кроме тестируемых библиотек, используются только встроенные модули Node.js.

7. Потенциальные точки расширения

  • Добавление новых парсеров: например, @fastify/busboy, multer (хотя multer работает на уровне middleware, его можно тестировать через интеграцию с Express).
  • Изменение профиля нагрузки: варьировать количество файлов, их размеры, добавлять смешанные данные (поля + файлы).
  • Измерение других метрик: пиковое использование памяти, скорость в МБ/с, влияние размера чанков при записи.
  • Параметризация через аргументы: передавать размеры файлов, количество повторов для усреднения.
  • Вывод результатов в табличном виде: вместо простого console.time можно собирать данные и выводить сравнительную таблицу.
  • Тестирование с разными версиями Node.js: для оценки влияния версии движка на производительность.
  • Добавление измерения нагрузки на дисковую подсистему: если парсеры сохраняют файлы на диск (как formidable по умолчанию).

8. Особенности для тестирования

  • Установка зависимостей: перед запуском убедитесь, что все три библиотеки установлены: 
    npm install busboy formidable multiparty
  • Запуск бенчмарка: 
    node bench-multipart-files-100mb-big.js busboy
node bench-multipart-files-100mb-big.js formidable
node bench-multipart-files-100mb-big.js multiparty
  • Ожидаемый вывод: для каждого запуска будет показано время выполнения, например: 
    busboy: 345.678ms
  • Проверка корректности подсчёта: после завершения можно вывести содержимое объекта calls (добавив console.log(calls)), чтобы убедиться, что все файлы были обработаны. Например, partBegin должно равняться количеству файлов (5), а partData должно быть значительным (зависит от количества чанков).
  • Ошибки при запуске: если библиотека не найдена, скрипт завершится с кодом 1 и сообщением об ошибке.
  • Влияние окружения: производительность может сильно зависеть от мощности процессора, скорости памяти и версии Node.js. Для достоверных сравнений запускайте несколько раз и усредняйте результаты.
  • Мониторинг памяти: можно запускать с флагом --trace-gc для анализа сборки мусора при обработке больших файлов.
