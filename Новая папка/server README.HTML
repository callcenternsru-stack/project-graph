2. Структура и семантика

Код организован как класс BlobsServer, реализующий полный цикл обработки HTTP-запросов для эмуляции Netlify Blobs:

  • Конструктор — принимает конфигурацию: директория для хранения, порт, токен доступа, функции логирования и отладки.
  • Маршрутизация — метод handleRequest направляет запросы в зависимости от HTTP-метода (GET, PUT, DELETE, HEAD).
  • Аутентификация — метод validateAccess проверяет Bearer-токен или подпись в URL.
  • Парсинг URL — поддержка двух форматов API (новый /api/v1/blobs/... и legacy /api/v1/sites/.../blobs).
  • Файловая структура:
    • {directory}/entries/{siteID}/{storeName}/{key} — данные блобов
    • {directory}/metadata/{siteID}/{storeName}/{key} — метаданные блобов
  • Потоковая обработка — при PUT запросе данные сохраняются во временный файл, затем атомарно копируются в целевой путь.
  • Листинг — поддержка параметров prefix и directories для фильтрации результатов.

3. Стилизация (CSS) — не применимо

Данный файл является серверным JavaScript-модулем и не содержит элементов пользовательского интерфейса. Весь код относится к бизнес-логике и обработке запросов.

4. Логика и интерактивность (JavaScript)

  • Обработка ошибок: используется функция isNodeError для корректной типизации ошибок Node.js, проверяются коды ENOENT, EISDIR.
  • Потоки (Streams): при GET-запросе данные отдаются через createReadStream с обработкой ошибок; при PUT-запросе входящий поток направляется во временный файл через pipeline.
  • Атомарные операции: запись выполняется во временную директорию, затем файл копируется в целевое место, что предотвращает повреждение данных при сбоях.
  • Поддержка подписанных URL: при наличии заголовка accept: application/json+netlify-blobs-signed-url или использовании legacy API возвращается подписанный URL вместо данных.
  • HMAC-подпись: токен доступа преобразуется в HMAC-SHA256 хеш и используется для создания подписанных URL.
  • Кроссплатформенность: учитывается особенности Windows (platform === "win32") при кодировании имен store в URL.
  • Рекурсивный обход: статический метод walk рекурсивно обходит директории для сбора списка блобов.
  • Генерация ETag: для каждого блоба создается случайный ETag (в реальном хранилище обычно используется хеш содержимого).

5. Форма и отправка данных

Сервер не имеет пользовательских форм. Данные передаются через HTTP-запросы:

  • PUT — тело запроса содержит бинарные данные блоба, метаданные передаются в заголовке X-Netlify-Blobs-Metadata.
  • GET — возвращает бинарные данные блоба или JSON со списком блобов.
  • DELETE — удаляет блоб и его метаданные.
  • HEAD — возвращает только метаданные (в заголовках).

Метаданные кодируются в base64 и передаются в специальном заголовке (METADATA_HEADER_INTERNAL).

6. Особенности реализации

  • Изоляция данных: данные и метаданные хранятся в отдельных директориях для упрощения резервного копирования и восстановления.
  • Поддержка регионов: URL может содержать префикс region:, который игнорируется при построении локальных путей.
  • Совместимость с Netlify: реализованы оба формата API (новый и legacy) для обратной совместимости.
  • Безопасность: если передан токен, сервер проверяет либо Bearer-токен в заголовке Authorization, либо подпись в query-параметре signature.
  • Отладка: при включенном режиме debug все запросы и ошибки логируются через переданный logger.
  • Гибкость: параметр onRequest позволяет вызывать внешнюю функцию при каждом запросе (для мониторинга).
  • Временные файлы: используются для атомарной записи, автоматически удаляются после копирования.
  • Обработка директорий: если при GET-запросе путь указывает на директорию, возвращается ошибка 404 (кроме случаев листинга).

7. Потенциальные точки расширения

  • Поддержка сжатия: добавить gzip/brotli для больших блобов.
  • Кэширование: реализовать in-memory кэш для часто запрашиваемых блобов.
  • Валидация метаданных: проверка размера метаданных (ограничение Netlify — 64KB).
  • Метрики и мониторинг: сбор статистики запросов (количество, размеры, время ответа).
  • Поддержка range-запросов: частичная загрузка файлов.
  • Репликация: возможность синхронизации с удаленным хранилищем.
  • WebDAV интерфейс: для удобного монтирования как диска.
  • Административный интерфейс: веб-панель для просмотра и управления блобами.

8. Особенности для тестирования

  • Запуск сервера: создайте экземпляр BlobsServer с параметрами и вызовите start(). 
    const { BlobsServer } = require('@netlify/blobs/dist/server.js');
const server = new BlobsServer({
    directory: './blobs-data',
    port: 3000,
    token: 'my-secret-token',
    debug: true
});
await server.start();
  • Проверка доступа: используйте заголовок Authorization: Bearer my-secret-token или добавьте параметр ?signature=... (хеш токена).
  • PUT запрос: отправьте бинарные данные на /api/v1/blobs/site123/store456/myblob с заголовком X-Netlify-Blobs-Metadata: eyJ... (base64).
  • GET запрос: запросите тот же URL — должны вернуться данные. Добавьте заголовок Accept: application/json+netlify-blobs-signed-url для получения подписанного URL.
  • Листинг: GET /api/v1/blobs/site123/store456/?prefix=my вернет JSON со списком блобов.
  • HEAD запрос: проверит существование блоба и вернет метаданные в заголовках.
  • DELETE запрос: удалит блоб.
  • Тестирование ошибок:
    • Запрос без токена → 403
    • Запрос к несуществующему блобу → 404
    • PUT без ключа → 400
    • Неподдерживаемый метод → 405
  • Проверка файловой системы: после PUT в ./blobs-data/entries/site123/store456/myblob должен появиться файл с данными, а в ./blobs-data/metadata/site123/store456/myblob — JSON с метаданными.
  • Нагрузочное тестирование: проверьте работу с большими файлами (более 100 МБ) и множеством одновременных запросов.
