📄 Инструкция: netlify.toml

Конфигурационный файл для платформы Netlify, определяющий настройки сборки, развёртывания и окружения.

Файл расположен: C:\Users\Контакт-Сервис\Desktop\prodect\netlify.toml

1. Базовые технические параметры

ПараметрЗначение
Тип документаTOML (Tom's Obvious Minimal Language) – формат конфигурационных файлов, удобный для чтения человеком
Кодировка символовUTF-8 (без BOM)
Роль в проектеУправляет процессом сборки и деплоя на Netlify, задаёт переменные окружения, указывает пути к функциям и статическим файлам
РедактированиеМожно и нужно редактировать вручную для изменения параметров деплоя
Взаимосвязь с другими файлами
  • Используется вместе с package.json – указывает, какие зависимости устанавливать и где находятся функции.
  • Переменные окружения, заданные в файле (NETLIFY_SITE_ID, NETLIFY_ACCESS_TOKEN), используются функциями для работы с Netlify Blobs.
  • Не имеет прямого отношения к клиентским HTML-страницам, но влияет на их доступность через настройки публикации.
  • В идеале должен находиться в корне репозитория.

2. Структура и семантика

Файл netlify.toml организован в виде секций (заголовков в квадратных скобках) и пар ключ = значение. В предоставленном файле присутствует только одна секция:

[build.environment]
NETLIFY_SITE_ID = "grafic1"
NETLIFY_ACCESS_TOKEN = "nfp_zJunGW2Sp99uRfNhSYmmo4Tyui1YwB853381"

Здесь:

  • [build.environment] – секция, определяющая переменные окружения, доступные на этапе сборки (build time).
  • NETLIFY_SITE_ID – идентификатор сайта на Netlify (используется для аутентификации при работе с Blobs).
  • NETLIFY_ACCESS_TOKEN – персональный токен доступа к API Netlify (тоже для работы с Blobs).

В типичном проекте также присутствуют секции:

  • [build] – основные настройки сборки: publish (папка с готовыми файлами), functions (папка с функциями).
  • [functions] – дополнительные настройки для функций (например, node_bundler = "esbuild").
  • [[redirects]] – правила перенаправления.
  • [[headers]] – пользовательские заголовки HTTP.

3. Отсутствие клиентских технологий

Поскольку netlify.toml – серверный конфигурационный файл, он не содержит CSS, JavaScript, HTML-форм и не отвечает за визуальное оформление. Все нижеследующие разделы стандартного шаблона здесь не применимы, но для полноты картины они кратко обозначены.

6. Особенности реализации

  • Безопасность: переменные NETLIFY_SITE_ID и NETLIFY_ACCESS_TOKEN являются чувствительными данными. Хранить их в открытом виде в репозитории не рекомендуется. Лучше задавать их через UI Netlify (в разделе Site settings > Environment variables) или использовать секреты в CI/CD. В текущем виде токен скомпрометирован – его следует немедленно отозвать и заменить.
  • Приоритет переменных: переменные, заданные в UI Netlify, имеют более высокий приоритет, чем указанные в netlify.toml.
  • Синтаксис TOML: строки заключаются в двойные кавычки. Ключи могут содержать точки (например, build.publish).
  • Отсутствие обязательных секций: даже если файл содержит только [build.environment], Netlify всё равно сможет развернуть проект, используя разумные значения по умолчанию (например, папка публикации – корень репозитория).
  • Необходимость передеплоя: после изменения netlify.toml нужно перезапустить деплой (автоматически при пуше в репозиторий или вручную).

7. Потенциальные точки расширения

  • Добавление секции [build]: 
    [build]
publish = "public"
functions = "netlify/functions"
    Указывает, что статические файлы находятся в папке public, а функции – в netlify/functions.
  • Настройка бандлинга функций: 
    [functions]
node_bundler = "esbuild"
    Ускоряет сборку функций.
  • Перенаправления (redirects): 
    [[redirects]]
from = "/api/*"
to = "/.netlify/functions/:splat"
status = 200
  • Переменные окружения для разных веток: 
    [context.production.environment]
NODE_ENV = "production"

[context.deploy-preview.environment]
NODE_ENV = "staging"
  • Заголовки безопасности: 
    [[headers]]
for = "/*"
[headers.values]
X-Frame-Options = "DENY"
X-Content-Type-Options = "nosniff"

8. Особенности для тестирования

  • Валидация синтаксиса: используйте онлайн-валидатор TOML (например, toml-lint.com) для проверки отсутствия ошибок.
  • Проверка переменных окружения: после деплоя зайдите в логи сборки на Netlify и убедитесь, что переменные NETLIFY_SITE_ID и NETLIFY_ACCESS_TOKEN доступны (они не выводятся в логи по соображениям безопасности, но можно проверить их наличие, добавив в функции вывод console.log(process.env.NETLIFY_SITE_ID) – но это тоже небезопасно).
  • Локальное тестирование с Netlify CLI: установите Netlify CLI и выполните netlify dev. CLI автоматически прочитает netlify.toml и подставит переменные окружения. Проверьте, что функции, использующие Blobs, работают корректно.
  • Безопасность токена: если токен был случайно опубликован, срочно сгенерируйте новый в настройках Netlify (User settings > Applications > Personal access tokens) и замените его в UI Netlify, удалив из файла.
  • Проверка путей: если добавите секцию [build] с publish и functions, убедитесь, что указанные папки существуют в репозитории.
