📄 Инструкция: .gitignore

Файл, указывающий Git, какие файлы и папки следует игнорировать при отслеживании версий.

Файл расположен: C:\Users\Контакт-Сервис\Desktop\prodect\.gitignore

1. Базовые технические параметры

ПараметрЗначение

  • Работает совместно с системой контроля версий Git.
  • Не зависит от других файлов проекта, но влияет на то, какие файлы будут отслеживаться.
  • Правила игнорирования применяются ко всем файлам в репозитории, включая конфигурационные (netlify.toml, package.json) и служебные (deno.lock, package-lock.json).
  • Особенно важен для игнорирования папки .netlify, которая создаётся локально и содержит временные данные.

2. Структура и семантика

Файл .gitignore состоит из строк, каждая из которых задаёт шаблон (паттерн) для игнорирования. Поддерживаются следующие синтаксические конструкции:

  • Комментарии – строки, начинающиеся с #. Git игнорирует их. Используются для пояснений.
  • Пустые строки – игнорируются, можно использовать для группировки.
  • Имена файлов/папок – например, .netlify означает, что папка с таким именем будет игнорироваться.
  • Слэш (/) – используется для указания относительных путей. Если путь начинается со слэша, он применяется относительно корня репозитория.
  • Звёздочка (*) – соответствует любому количеству символов, кроме слэша.
  • Двойная звёздочка (**) – соответствует любому количеству символов, включая слэши (используется для рекурсивного игнорирования).
  • Восклицательный знак (!) – инвертирует правило (файл не игнорируется, даже если подходит под предыдущий паттерн).

В предоставленном файле содержится:

# Local Netlify folder
.netlify

Здесь:

  • # Local Netlify folder – комментарий, поясняющий назначение следующего правила.
  • .netlify – правило, указывающее Git игнорировать папку .netlify в корне проекта (и во всех подпапках, так как не указан начальный слэш).

3. Отсутствие клиентских технологий

Поскольку .gitignore – это служебный файл для системы контроля версий, он не содержит CSS, JavaScript, HTML-форм и не отвечает за визуальное оформление. Все нижеследующие разделы стандартного шаблона здесь не применимы, но для полноты картины они кратко обозначены.

4. JavaScript (логика и интерактивность)

6. Особенности реализации

  • Игнорирование уже отслеживаемых файлов: если файл уже был добавлен в репозиторий (commit), добавление его в .gitignore не заставит Git перестать его отслеживать. Для этого нужно удалить файл из индекса командой git rm --cached.
  • Папка .netlify: эта папка создаётся локально при использовании Netlify CLI (netlify dev, netlify link и т.д.). Она содержит временные файлы, кэш, конфигурации окружения, которые не должны попадать в репозиторий. Игнорирование этой папки – обязательная практика.
  • Глобальный .gitignore: помимо локального файла, Git поддерживает глобальный файл игнорирования (обычно ~/.gitignore_global), который применяется ко всем репозиториям на машине. В нём удобно задавать системные игнорирования (например, .DS_Store для macOS).
  • Регистр символов: правила в .gitignore чувствительны к регистру (на Linux-системах). На Windows и macOS регистр может игнорироваться, но для совместимости лучше придерживаться точного регистра.
  • Проверка игнорирования: команда git check-ignore -v <файл> показывает, какое правило привело к игнорированию данного файла.

7. Потенциальные точки расширения

  • Игнорирование системных файлов: добавить правила для файлов, создаваемых операционной системой или редакторами, например: 
    # macOS
.DS_Store

# Windows
Thumbs.db

# Редакторы
*.swp
*.swo
*~
  • Игнорирование зависимостей: обычно node_modules уже игнорируется, но можно добавить и другие директории с зависимостями (vendor/, bower_components/).
  • Игнорирование файлов с секретами: например, .env, .env.local, secrets.json.
  • Игнорирование логов и временных файлов: *.log, tmp/, cache/.
  • Игнорирование файлов сборки: dist/, build/, out/.
  • Исключения из правил: если нужно игнорировать все файлы определённого типа, но разрешить некоторые, можно использовать !, например: 
    *.log
!important.log

8. Особенности для тестирования

  • Проверка синтаксиса: можно использовать онлайн-инструменты или команду git check-ignore для проверки, какие файлы будут проигнорированы.
  • Тестирование правил: создайте тестовый файл, который должен игнорироваться, и выполните git status --ignored. Файл должен отображаться в разделе "Ignored files".
  • Воспроизводимость: убедитесь, что при клонировании репозитория на новую машину и выполнении git status не появляются неожиданные файлы (например, .netlify).
  • Проверка папки .netlify: если вы уже запускали netlify dev, папка .netlify должна присутствовать локально, но git status не должен её показывать. Если она отображается как untracked, значит правило в .gitignore не работает (возможно, файл уже был добавлен в репозиторий ранее).
  • Тестирование с разными Git-клиентами: убедитесь, что правила корректно работают как в консольном Git, так и в GUI-клиентах (GitHub Desktop, GitKraken и т.д.).
  • Команда для проверки: git ls-files --others --ignored --exclude-standard показывает все игнорируемые файлы.
