2. Структура и семантика

Скрипт предназначен для бенчмаркинга и сравнения трёх популярных библиотек для разбора multipart/form-data. Он генерирует синтетический запрос, состоящий из 100 полей по 1 МБ (общий объём около 100 МБ), и измеряет время, затраченное каждой библиотекой на полную обработку. Это позволяет оценить производительность при работе с большим количеством мелких полей.

  • Генерация данных: функция createMultipartBuffers создаёт буферы, соответствующие multipart-сообщению с заданными размерами полей. Каждое поле заполняется повторяющимися символами '0'.
  • Счётчик событий: объект calls подсчитывает количество вызовов различных этапов парсинга (начало части, заголовок, данные, конец). Это позволяет убедиться, что все библиотеки корректно обработали все 100 полей.
  • Выбор библиотеки: через аргумент командной строки передаётся имя модуля (busboy, formidable или multiparty). В зависимости от этого выполняется соответствующий блок кода.
  • Измерение времени: перед началом обработки вызывается console.time(moduleName), по окончании (событие close или end) — console.timeEnd(moduleName).

3. Стилизация (CSS) — не применимо

4. JavaScript (логика и интерактивность)

  • Синхронная генерация данных: все буферы создаются заранее.
  • Асинхронная обработка: парсеры работают в событийно-ориентированном стиле, данные передаются через вызовы write() или push().
  • Подсчёт событий:
    • Для busboy используется событие field, которое инкрементирует partBegin, partData, partEnd.
    • Для formidable используется событие data, где имя события (partBegin, partData и т.д.) инкрементирует соответствующий счётчик.
    • Для multiparty применяется «перехват» (hijack) внутренних методов onParseHeaderField, onParseHeaderValue, onParsePartBegin, onParsePartData, onParsePartEnd для увеличения счётчиков.
  • Обработка множества полей: 100 полей по 1 МБ, библиотеки должны работать с лимитами, установленными в Infinity (или аналогичными).
  • Массив размеров: (new Array(100)).fill(1) создаёт массив из 100 элементов со значением 1, что соответствует 100 полям по 1 МБ.

5. Форма и отправка данных

Скрипт не отправляет данные по сети; он симулирует получение multipart-запроса, передавая буферы напрямую парсеру. Это позволяет измерить чистое время парсинга без влияния сетевой задержки.

6. Особенности реализации

  • Гранулярный подсчёт событий: для каждого парсера используются разные методы подсчёта, чтобы получить сопоставимые данные о количестве обработанных частей.
  • Бесконечные лимиты: для busboy и multiparty устанавливаются fieldSizeLimit: Infinity (или аналоги), чтобы избежать обрезания полей.
  • Хитрый перехват в multiparty: поскольку multiparty не предоставляет прямых событий для каждого шага парсинга, скрипт временно заменяет внутренние методы, чтобы вести подсчёт.
  • Использование Readable stream для multiparty: создаётся фиктивный HTTP-запрос (new Readable), в который последовательно передаются буферы.
  • Отсутствие внешних зависимостей: кроме тестируемых библиотек, используются только встроенные модули Node.js.
  • Сравнение сценариев: этот бенчмарк дополняет "big" версию (с несколькими большими полями), проверяя производительность на множестве мелких полей.

7. Потенциальные точки расширения

  • Добавление новых парсеров: например, @fastify/busboy, multer.
  • Изменение профиля нагрузки: варьировать количество полей (50, 200, 500), их размеры (512 КБ, 2 МБ), добавлять смешанные типы.
  • Измерение других метрик: пиковое использование памяти, скорость в МБ/с, влияние размера чанков при записи.
  • Параметризация через аргументы: передавать количество полей и их размер через командную строку.
  • Вывод результатов в табличном виде: собирать данные и выводить сравнительную таблицу с усреднёнными значениями.
  • Тестирование с разными версиями Node.js: для оценки влияния версии движка на производительность.
  • Добавление измерения нагрузки на CPU: профилирование с использованием встроенных модулей.

8. Особенности для тестирования

  • Установка зависимостей: перед запуском убедитесь, что все три библиотеки установлены: 
    npm install busboy formidable multiparty
  • Запуск бенчмарка: 
    node bench-multipart-fields-100mb-small.js busboy
node bench-multipart-fields-100mb-small.js formidable
node bench-multipart-fields-100mb-small.js multiparty
  • Ожидаемый вывод: для каждого запуска будет показано время выполнения, например: 
    busboy: 234.567ms
  • Проверка корректности подсчёта: после завершения можно вывести содержимое объекта calls (добавив console.log(calls)), чтобы убедиться, что все 100 полей были обработаны (например, partBegin должно равняться 100).
  • Ошибки при запуске: если библиотека не найдена, скрипт завершится с кодом 1 и сообщением об ошибке.
  • Влияние окружения: производительность может сильно зависеть от мощности процессора и версии Node.js. Для достоверных сравнений запускайте несколько раз и усредняйте результаты.
  • Мониторинг памяти: можно запускать с флагом --trace-gc для анализа сборки мусора.
