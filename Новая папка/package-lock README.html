📄 Инструкция: package-lock.json

Файл блокировки версий зависимостей npm (автоматически генерируемый).

ПараметрЗначение

Кодировка символовUTF-8 (без BOM)
Роль в проектеФиксирует точные версии всех установленных пакетов и их зависимостей для обеспечения воспроизводимости сборки

  • Не требуется для работы клиентских страниц напрямую, но необходим при развёртывании на Netlify для корректной установки функций.

2. Структура и семантика

package-lock.json имеет корневой объект со следующими основными полями:

  • name – имя проекта (берётся из package.json).
  • version – версия проекта.
  • lockfileVersion – версия формата lock-файла (в примере 3, что соответствует npm v7+).
  • requires – булево значение, указывающее, используются ли точные версии.
  • packages – объект, содержащий информацию обо всех пакетах, установленных в node_modules. Ключи – пути к пакетам (например, "" для корневого проекта, "node_modules/@netlify/blobs" для конкретной зависимости).
  • dependencies – устаревшее поле (для обратной совместимости), содержит информацию о зависимостях в старом формате.

В разделе packages для каждого пакета указываются:

  • version – точная версия.
  • resolved – URL, откуда был загружен пакет (обычно из реестра npm).
  • integrity – хеш для проверки целостности (Subresource Integrity).
  • dependencies – объект с зависимостями данного пакета (если есть).
  • dev – флаг, указывающий, что пакет является разработческим (если установлен с --save-dev).
  • license – лицензия пакета (может отсутствовать).

3. Содержимое и зависимости

В вашем проекте используются следующие прямые зависимости (указаны в package.json):

  • @netlify/blobs версии ^7.4.0 (в lock-файле зафиксирована версия 7.4.0) – библиотека для работы с Netlify Blobs (используется в Netlify Functions).
  • busboy версии ^1.6.0 (зафиксирована версия 1.6.0) – парсер multipart/form-data, необходим для приёма файлов в функциях (например, submitResults.js).

Кроме того, busboy имеет свою зависимость streamsearch версии 1.1.0, которая также зафиксирована.

4. Использование и автоматизация

Файл package-lock.json используется следующими инструментами:

  • npm ci – команда для чистой установки зависимостей в CI-средах (Continuous Integration). Она использует lock-файл для установки точно указанных версий, игнорируя package.json. Это гарантирует, что сборка будет идентична разработческой.
  • npm install – при наличии lock-файла npm старается установить версии, соответствующие lock-файлу, и обновляет его, если package.json был изменён.
  • Netlify (при деплое) – при сборке проекта на Netlify автоматически выполняется npm ci, поэтому lock-файл необходим для воспроизводимой установки зависимостей функций.

5. Особенности реализации

  • Не редактировать вручную: любые изменения в lock-файле могут привести к конфликтам и некорректной работе. Все изменения зависимостей должны производиться через npm install <package> или редактирование package.json с последующим запуском npm install.
  • Хранение в репозитории: файл package-lock.json должен быть добавлен в систему контроля версий (Git), чтобы все члены команды и сборочные серверы использовали одни и те же версии.
  • Конфликты слияния: при возникновении конфликтов в lock-файле их нужно разрешать, переустановив зависимости (npm install) после слияния веток.
  • Связь с deno.lock: в проекте также присутствует deno.lock – это аналогичный файл для Deno, но он используется для edge-функций Netlify (если они написаны на Deno).

6. Потенциальные точки расширения

  • Добавление новых зависимостей: при необходимости добавить новую библиотеку (например, для валидации, логирования) нужно выполнить npm install <package>, что автоматически обновит package.json и package-lock.json.
  • Обновление существующих зависимостей: можно выполнить npm update для обновления в рамках семантического версионирования, либо установить конкретную версию через npm install <package>@<version>.
  • Удаление зависимостей: через npm uninstall <package>.
  • Использование yarn: если команда предпочитает yarn, будет создан yarn.lock. Но в проекте используется npm, поэтому package-lock.json остаётся основным.

7. Особенности для тестирования

  • Проверка целостности: выполните npm ci в чистой директории (без node_modules). Команда должна успешно выполниться и вывести отчёт об установленных пакетах.
  • Отсутствие ошибок: убедитесь, что после npm ci все функции проекта работают корректно (особенно те, что используют @netlify/blobs и busboy).
  • Проверка версий: можно вручную сверить версии в package-lock.json с теми, что реально устанавливаются – они должны совпадать.
  • Тестирование в изоляции: удалите папку node_modules и выполните npm ci – это займёт некоторое время, но должно завершиться без ошибок.
  • Конфликты версий: если после обновления каких-либо пакетов возникают ошибки в функциях, можно откатить версии, восстановив предыдущий lock-файл из репозитория и повторно выполнив npm ci.
